## 1. Введение и контекст

### 1.1 Назначение документа

Данный документ является основным техническим заданием для разработки системы AnticiFi. Он определяет функциональные и нефункциональные требования, технологический стек, архитектуру системы, а также критерии успешной реализации MVP. Документ предназначен для команды разработки, технических архитекторов и руководства продукта.

### 1.2 Продукт

AnticiFi — предиктивный AI-ассистент для управления личными финансами. Ключевое отличие от конкурентов: система смотрит вперёд, а не назад. Вместо анализа прошлых трат AnticiFi прогнозирует будущее состояние баланса пользователя на произвольную дату, учитывая паттерны расходов, запланированные платежи и сезонные факторы.

Слоган продукта: **"Know your balance before it happens"**

### 1.3 Цель MVP

Цель первого публичного релиза — валидация ключевой гипотезы: пользователи готовы доверять прогнозу баланса, сформированному на основе их финансовой истории, и принимать на его основе финансовые решения.

MVP должен обеспечить:

- Регистрацию пользователей и управление одним финансовым счётом (Free tier)
- Импорт транзакций через CSV и OCR-сканирование чеков
- Предсказание баланса на 30/60/90 дней вперёд
- Управление регулярными (запланированными) платежами
- Push-уведомления об угрозе дефицита баланса
- Мобильное приложение на Flutter (iOS + Android)

## 2. Область применения MVP

### 2.1 Включено в MVP

| Функциональная область | Описание |
|---|---|
| Аутентификация | Регистрация, вход, выход, refresh token |

- Экспорт данных в PDF/Excel
- B2B white-label API
- Web-приложение (только мобильное в MVP)
- Prometheus/Grafana мониторинг (настройка инфраструктуры, но без дашбордов)

## 3. Технологический стек

### 3.1 Backend

#### Runtime: Node.js 20 LTS

**Обоснование выбора:**

Node.js 20 LTS — долгосрочная поддержка до апреля 2026 года с последующим продлением до 2027. Выбор обусловлен несколькими факторами:

- **Единый язык с фронтом**: TypeScript используется как на backend, так и в потенциальном web-frontend, что снижает когнитивную нагрузку на команду и упрощает переиспользование типов и валидационных схем.
- **Async I/O**: Событийно-ориентированная архитектура Node.js идеально подходит для паттернов высокой конкурентности — обработки множества одновременных WebSocket-соединений и NATS-сообщений без блокировки потока.
- **Экосистема npm**: Богатая экосистема готовых библиотек (Bull, Socket.IO, Sequelize, JWT, bcrypt) позволяет не изобретать велосипед в инфраструктурных задачах.
- **Производительность**: При I/O-bound нагрузке (характерной для API Gateway и микросервисов) Node.js конкурентоспособен с Go и значительно превосходит Python/Ruby в throughput.

#### Framework: NestJS

**Обоснование выбора:**

NestJS — единственный enterprise-grade framework для Node.js с полноценной поддержкой микросервисной архитектуры из коробки.

- **Модульность и Dependency Injection**: Архитектура, вдохновлённая Angular, обеспечивает чёткое разделение ответственности. Каждый модуль инкапсулирует свои контроллеры, сервисы и репозитории. DI-контейнер упрощает тестирование через mock-объекты.
- **TypeScript-first**: Полная поддержка декораторов и строгой типизации. Декораторы `@Controller`, `@Injectable`, `@MessagePattern` делают код самодокументируемым.
- **Встроенная поддержка NATS**: NestJS имеет официальный транспортный адаптер для NATS через `@nestjs/microservices`, что упрощает создание NATS-консьюмеров в несколько строк кода.
- **Guards, Interceptors, Pipes**: Встроенная pipeline для аутентификации (JwtGuard), трансформации данных (ClassSerializerInterceptor) и валидации (ValidationPipe) позволяет решать cross-cutting concerns декларативно.
- **Swagger-интеграция**: `@nestjs/swagger` автоматически генерирует OpenAPI-документацию из декораторов, устраняя необходимость поддерживать документацию вручную.

#### ORM: Sequelize v6

**Обоснование выбора:**

- **Зрелость**: Sequelize существует с 2011 года и является одним из самых battle-tested ORM для Node.js. Это критично для production-систем с финансовыми данными.
- **Миграции**: Встроенная система миграций через `sequelize-cli` обеспечивает версионирование схемы базы данных, воспроизводимость окружений и безопасные схемные изменения без downtime (через транзакционные миграции).
- **PostgreSQL-совместимость**: Полная поддержка специфических типов PostgreSQL (JSONB, ARRAY, UUID, ENUM), частичных индексов и `ON CONFLICT DO UPDATE` (upsert).
- **Хуки (Hooks)**: Lifecycle hooks (`beforeCreate`, `afterUpdate`, `beforeDestroy`) позволяют реализовывать бизнес-логику на уровне модели — например, автоматическое обновление баланса счёта при создании транзакции, или публикацию события в NATS после сохранения.
- **Мягкое удаление (Paranoid)**: Встроенная поддержка soft delete через `paranoid: true` — транзакции помечаются как удалённые вместо физического удаления, что важно для аудита и compliance.

#### Database: PostgreSQL 16

**Обоснование выбора:**

- **Надёжность и ACID**: PostgreSQL — эталон надёжности в мире RDBMS. Полная поддержка транзакций ACID критична для финансовых операций, где потеря или дублирование данных неприемлемы.
- **JSONB**: Возможность хранить метаданные транзакций (raw-данные из CSV, OCR-результаты) в JSONB-полях с возможностью индексирования и запросов к содержимому. Это устраняет необходимость в отдельной NoSQL-базе для неструктурированных данных.
- **Полнотекстовый поиск**: Встроенный full-text search через `tsvector`/`tsquery` позволяет реализовать поиск по описаниям транзакций без внешнего поискового движка (Elasticsearch) на этапе MVP.
- **Партиционирование**: PostgreSQL 16 поддерживает нативное партиционирование таблиц по диапазону дат. Таблица `transactions`, достигающая 100M+ записей, может быть партиционирована по году/месяцу для сохранения производительности запросов.
- **Read Replicas**: Streaming replication позволяет настроить читающие реплики для масштабирования чтения без изменения кода приложения.
- **Row-Level Security**: Механизм RLS позволяет принудительно изолировать данные пользователей на уровне базы данных — дополнительный уровень защиты в multi-tenant архитектуре.

#### Messaging: NATS

**Обоснование выбора:**

NATS — высокопроизводительная система обмена сообщениями, написанная на Go. Выбрана в противовес RabbitMQ и Apache Kafka.

- **Легковесность**: Сервер NATS запускается за миллисекунды, потребляет минимальные ресурсы памяти (~10MB в покое), что критично для локальной разработки через Docker Compose.
- **Производительность**: NATS способен обрабатывать десятки миллионов сообщений в секунду с задержкой < 1ms. Для масштабов MVP и даже Series A это с многократным запасом.
- **Паттерн Request/Reply**: В отличие от Kafka (только pub/sub), NATS нативно поддерживает синхронный request/reply — это позволяет api-gateway запрашивать данные у микросервисов как через RPC, не поднимая отдельный gRPC-стек.
- **Pub/Sub для событий**: Одновременная поддержка pub/sub позволяет реализовывать event-driven архитектуру — после создания транзакции api-gateway публикует событие, которое независимо потребляют notification-service и prediction-service.
- **JetStream**: Встроенный persistent messaging через JetStream обеспечивает at-least-once delivery для критических событий (например, импорт файла) без необходимости в отдельном брокере.

#### Queue: Bull (Redis-backed)

**Обоснование выбора:**

- **Надёжность задач**: Bull обеспечивает персистентные очереди задач на базе Redis. В случае падения сервиса задача не теряется — она остаётся в Redis и будет обработана при рестарте.
- **Retry и Backoff**: Встроенная поддержка exponential backoff при повторных попытках. Это критично для OCR-обработки и внешних API-вызовов, которые могут временно быть недоступны.
- **Delayed Jobs**: Возможность запланировать задачу на конкретное время — например, отправить push-уведомление за 3 дня до ожидаемого дефицита баланса.
- **Concurrency Control**: Настраиваемое количество параллельных воркеров. OCR-обработка — вычислительно интенсивная задача, для неё устанавливается ограничение в 2-3 параллельных воркера.
- **Bull Board**: Готовый UI для мониторинга состояния очередей в режиме разработки.

#### Cache: Redis 7

**Обоснование выбора:**

Redis используется в трёх независимых ролях в системе:

1. **Кеширование**: Результаты ML-предсказаний кешируются на 1 час. Прогноз для конкретного пользователя пересчитывается редко (только при новых транзакциях), поэтому кеш значительно снижает нагрузку на Python-сервис.
2. **Хранение сессий и refresh tokens**: Refresh tokens хранятся в Redis с TTL 7 дней. При logout токен немедленно инвалидируется — это невозможно реализовать с stateless JWT без blacklist.
3. **Pub/Sub для Socket.IO**: При горизонтальном масштабировании api-gateway несколько инстансов должны синхронизировать WebSocket-комнаты. Redis adapter для Socket.IO использует pub/sub для трансляции событий между инстансами.

#### Real-time: Socket.IO

**Обоснование выбора:**

- **WebSocket с fallback**: Socket.IO автоматически деградирует до long-polling при недоступности WebSocket, что критично для надёжности на мобильных сетях.
- **Rooms и Namespaces**: Пространства имён (`/notifications`, `/dashboard`) и комнаты (по user ID) позволяют адресовать события конкретному пользователю или группе без broadcast всем.
- **Переподключение**: Автоматическое переподключение с exponential backoff — мобильное приложение не требует ручной логики reconnect.
- **Redis Adapter**: Официальный адаптер `@socket.io/redis-adapter` обеспечивает работу в кластере с несколькими инстансами сервера.

#### Auth: JWT + bcrypt

**Обоснование выбора:**

- **JWT (Access + Refresh)**: Двухтокенная схема с короткоживущим access token (15 минут) и долгоживущим refresh token (7 дней). Access token stateless — проверяется без обращения к базе. Refresh token хранится в Redis — это позволяет немедленно отозвать сессию при компрометации.
- **bcrypt (12 rounds)**: Адаптивная хеш-функция, специально разработанная для хеширования паролей. 12 раундов обеспечивают ~300ms на современном сервере — достаточно медленно для защиты от brute-force, достаточно быстро для UX.

### 3.2 AI/ML Сервис

#### Python 3.11 + FastAPI

**Обоснование:**

Python — стандарт де-факто для ML/Data Science экосистемы. FastAPI выбран за:
- Асинхронную поддержку через `asyncio` — обработка нескольких prediction-запросов параллельно.
- Автоматическую генерацию OpenAPI-документации.
- Pydantic-валидацию входных данных.

#### Модели предсказания

Система использует ансамблевый подход:

| Модель | Назначение | Условие применения |
|---|---|---|
| Prophet (Meta) | Предсказание временных рядов с сезонностью | Пользователи с историей > 3 месяцев |
| LSTM (кастомный) | Обнаружение паттернов в нерегулярных тратах | Пользователи с историей > 6 месяцев |
| scikit-learn (linear) | Базовое предсказание | Новые пользователи, < 3 месяцев истории |

Выбор модели происходит автоматически на основе объёма доступной истории транзакций.

#### Коммуникация с backend

AI-сервис интегрируется в систему через NATS — он подписывается на subject `prediction.generate` и возвращает результат через request/reply. Это изолирует Python-процесс от NestJS-инфраструктуры и позволяет независимо масштабировать и деплоить сервис предсказаний.

### 3.3 Мобильное приложение

#### Flutter 3.x + Dart

**Обоснование:**

- **Cross-platform**: Один кодовой базы для iOS и Android — критично при ограниченных ресурсах стартапа.
- **Производительность**: Компиляция в нативный ARM-код, собственный rendering engine (Skia/Impeller) — плавные анимации без JavaScript bridge.
- **Hot Reload**: Мгновенное обновление UI при разработке.

#### State Management: BLoC/Cubit

Архитектурный паттерн BLoC (Business Logic Component) обеспечивает строгое разделение UI и бизнес-логики. Cubit — упрощённая версия BLoC без event-объектов, используется для простых состояний (загрузка, ошибка, данные).

Подробная документация Flutter-архитектуры — в [Мобильном приложении](./mobile-app.md).

#### Локальная БД: Isar

Isar — высокопроизводительная NoSQL база для Flutter, написанная на Rust. Используется для:
- Хранения последнего прогноза в офлайн-режиме.
- Кеширования списка транзакций для мгновенного отображения без сетевого запроса.
- Хранения draft-записей транзакций до синхронизации с сервером.

#### OCR: Google ML Kit

Google ML Kit Text Recognition v2 работает полностью на устройстве (on-device inference) — не требует интернет-соединения и не передаёт изображения чека в облако, что критично для приватности финансовых данных.

#### Navigation: GoRouter

GoRouter — официально рекомендованная навигационная библиотека Flutter. Поддерживает декларативную маршрутизацию, deep links и защищённые маршруты (redirect при отсутствии аутентификации).

#### HTTP: Dio

Dio поддерживает interceptors — используется для автоматического добавления Authorization-заголовка и автоматического обновления access token через refresh при получении 401.

### 4.1 Топология сервисов

AnticiFi реализует микросервисную архитектуру с асинхронным обменом сообщениями через NATS. Все сервисы развёртываются как отдельные Docker-контейнеры.

```
┌─────────────────────────────────────────────────────────────────┐
│                        MOBILE APP (Flutter)                      │
│                    iOS / Android                                  │
└─────────────────────┬──────────────────────┬────────────────────┘
                      │ REST / HTTPS          │ WebSocket (Socket.IO)
                      ▼                       ▼
┌─────────────────────────────────────────────────────────────────┐
│                         NGINX                                     │
│              Reverse Proxy / SSL Termination                      │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                       API GATEWAY (NestJS)                        │
│   Auth │ REST Controllers │ WebSocket Gateway │ Bull Queues       │
└────────────────────────────┬────────────────────────────────────┘
                              │ NATS publish / request
          ┌───────────────────┼───────────────────┐
          ▼                   ▼                   ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────────┐
│ TRANSACTION     │ │ IMPORT SERVICE  │ │ NOTIFICATION        │
│ SERVICE         │ │ (NestJS)        │ │ SERVICE (NestJS)     │
│ (NestJS)        │ │ CSV + OCR Queue │ │ Push + Email        │
└────────┬────────┘ └────────┬────────┘ └─────────────────────┘
         │                   │
         │         ┌─────────┴──────────┐
         │         ▼                    ▼
         │ ┌────────────────┐ ┌────────────────┐
         │ │   BULL (Redis) │ │ PREDICTION SVC  │
         │ │   Job Queues   │ │ (Python/FastAPI)│
         │ └────────────────┘ │ Prophet / LSTM  │
         │                    └────────┬───────┘
         │                             │
         ▼                             ▼
┌─────────────────────────────────────────────────────────────────┐
│                    POSTGRESQL 16                                   │
│         Users │ Accounts │ Transactions │ Predictions            │
└─────────────────────────────────────────────────────────────────┘
                                │
                    ┌───────────┴───────────┐
                    ▼                       ▼
         ┌──────────────────┐   ┌──────────────────┐
         │    REDIS 7        │   │      NATS         │
         │  Cache / Sessions │   │  Message Broker   │
         │  Bull Backend    │   │  Request/Reply     │
         └──────────────────┘   └──────────────────┘
```

### 4.2 Описание сервисов

| Сервис | Технология | Роль |
|---|---|---|
| `api-gateway` | NestJS | Единая точка входа. REST API, JWT Auth, WebSocket |
| `transaction-service` | NestJS | NATS-консьюмер. Хранение и обработка транзакций |
| `import-service` | NestJS | Парсинг CSV, оркестрация OCR-задач, дедупликация |
| `prediction-service` | Python/FastAPI | ML-модели. Генерация прогнозов баланса |
| `notification-service` | NestJS | Управление и доставка push-уведомлений и email |

### 4.3 Принципы взаимодействия

**Синхронная коммуникация (Request/Reply через NATS):**

Используется когда api-gateway ожидает ответ от микросервиса в рамках одного HTTP-запроса. Например: получить список транзакций, запросить актуальный прогноз.

```
API Gateway  ──nats.request('transaction.findAll', payload)──▶  Transaction Service
             ◀────────────────────────── response ────────────────────────────────
```

**Асинхронная коммуникация (Pub/Sub через NATS):**

Используется для событий, где api-gateway не ожидает ответа. Несколько сервисов могут обработать одно событие независимо.

```
API Gateway  ──nats.publish('transaction.created', event)──▶  Prediction Service
                                                          ──▶  Notification Service
```

**Очереди задач (Bull через Redis):**

Используется для тяжёлых фоновых задач с retry и мониторингом: парсинг CSV-файлов, OCR-обработка изображений, массовая рассылка уведомлений.

## 5. Описание модулей

### 5.1 AuthModule

**Ответственность:**

Управление жизненным циклом аутентификации пользователей: регистрация, вход, выход, обновление токенов. Является единственным модулем, который обращается к таблице `users` для операций с паролями.

**Ключевые endpoints:**

| Метод | Путь | Описание | Auth |
|---|---|---|---|
| POST | `/auth/register` | Регистрация нового пользователя | Нет |
| POST | `/auth/login` | Вход, получение токен-пары | Нет |
| POST | `/auth/refresh` | Обновление access token по refresh token | Нет |
| POST | `/auth/logout` | Инвалидация refresh token | Access JWT |
| GET | `/auth/me` | Получение профиля текущего пользователя | Access JWT |

**Ключевая бизнес-логика:**

1. **Регистрация**: Проверка уникальности email → хеширование пароля (bcrypt, 12 rounds) → сохранение пользователя → генерация токен-пары → запись refresh token в Redis с TTL 7 дней → возврат токенов клиенту.

2. **Вход**: Поиск пользователя по email → сравнение хеша пароля → генерация новой токен-пары → запись refresh token в Redis → возврат токенов.

3. **Refresh**: Верификация refresh token из Redis (не только JWT-подпись, но и наличие в Redis) → генерация новой токен-пары → ротация: старый refresh token удаляется из Redis, новый записывается → возврат новой пары. Rotation предотвращает replay-атаки на украденный refresh token.

4. **Logout**: Удаление refresh token из Redis — с этого момента токен считается недействительным, даже если его подпись валидна.

**Зависимости:**

- `UsersModule` (для поиска пользователя по email)
- Redis (для хранения refresh tokens)
- `@nestjs/jwt` (генерация и верификация JWT)
- `bcrypt` (хеширование паролей)

### 5.2 UserModule

**Ответственность:**

Управление профилем пользователя и пользовательскими настройками. Модуль не занимается аутентификацией — только CRUD профиля.

### 5.3 AccountModule

**Ответственность:**

Управление финансовыми счетами пользователя. В MVP — один счёт на пользователя (Free tier). Счёт является корневым агрегатом для транзакций и прогнозов.

**Ключевые endpoints:**

| Метод | Путь | Описание | Auth |
|---|---|---|---|
| POST | `/accounts` | Создание счёта | Access JWT |
| GET | `/accounts` | Список счетов пользователя | Access JWT |
| GET | `/accounts/:id` | Детали счёта | Access JWT |
| PATCH | `/accounts/:id` | Обновление счёта (имя, начальный баланс) | Access JWT |
| DELETE | `/accounts/:id` | Удаление счёта (с транзакциями) | Access JWT |
| GET | `/accounts/:id/balance` | Текущий баланс с учётом транзакций | Access JWT |

**Ключевая бизнес-логика:**

- **Баланс**: Текущий баланс счёта вычисляется динамически — `initialBalance + SUM(income transactions) - SUM(expense transactions)`. Это значение кешируется в Redis с TTL 5 минут и инвалидируется при создании новой транзакции.
- **Ограничение Free Tier**: При попытке создания второго счёта возвращается 403 с кодом ошибки `UPGRADE_REQUIRED`. Проверка происходит через `SubscriptionGuard`.
- **Currency**: Счёт хранит валюту (ISO 4217). Все суммы транзакций должны совпадать с валютой счёта. Конвертация валют — Post-MVP функциональность.

**Зависимости:**

- PostgreSQL (таблица `accounts`)
- Redis (кеш баланса)
- `TransactionModule` (для расчёта баланса)
- `SubscriptionModule` (проверка tier при создании)

### 5.4 TransactionModule

**Ответственность:**

Центральный модуль системы. Хранение, категоризация и агрегация финансовых транзакций. Является единственным источником истины для расчёта баланса и входным источником данных для ML-моделей.

**Ключевые endpoints (REST):**

| Метод | Путь | Описание | Auth |
|---|---|---|---|
| POST | `/transactions` | Ручное создание транзакции | Access JWT |
| GET | `/transactions` | Список с фильтрацией и пагинацией | Access JWT |
| GET | `/transactions/:id` | Детали транзакции | Access JWT |
| PATCH | `/transactions/:id` | Редактирование (сумма, категория, дата) | Access JWT |
| DELETE | `/transactions/:id` | Мягкое удаление | Access JWT |
| GET | `/transactions/stats` | Агрегированная статистика по периоду | Access JWT |

**NATS subjects:**

| Subject | Тип | Описание |
|---|---|---|
| `transaction.create` | Request/Reply | Создание транзакции из import-service |
| `transaction.findAll` | Request/Reply | Получение транзакций для ML-модели |
| `transaction.created` | Publish | Событие о создании — для prediction, notification |
| `transaction.stats` | Request/Reply | Агрегированная статистика по аккаунту |

**Ключевая бизнес-логика:**

- **Категоризация**: При создании транзакции без явной категории запускается автоматическая классификация на основе ключевых слов в описании (rule-based в MVP, ML-based — Post-MVP).
- **Дедупликация**: При импорте из CSV используется составной ключ `(accountId, date, amount, description)` для предотвращения дублирования транзакций при повторном импорте того же файла.
- **Инвалидация кеша**: После каждой записи/изменения транзакции автоматически инвалидируется кеш баланса счёта в Redis через Sequelize-hook `afterCreate` / `afterUpdate` / `afterDestroy`.
- **Паджинация**: Курсорная пагинация (cursor-based) вместо offset-пагинации для устойчивой работы с таблицей 100M+ записей. Курсор — UUID последней записи.

**Зависимости:**

- PostgreSQL (таблица `transactions`, `categories`)
- Redis (инвалидация кеша баланса)
- NATS (pub/sub событий, request/reply от других сервисов)

### 5.5 ImportModule

**Ответственность:**

Обработка импорта транзакций из внешних источников: банковские CSV-выписки и фотографии бумажных чеков через OCR. Модуль оркестрирует фоновые задачи через Bull Queue.

**Ключевые endpoints:**

| Метод | Путь | Описание | Auth |
|---|---|---|---|
| POST | `/import/csv` | Загрузка CSV-файла | Access JWT |
| GET | `/import/:jobId/status` | Статус задачи импорта | Access JWT |
| POST | `/import/ocr` | Отправка изображения чека | Access JWT |
| GET | `/import/ocr/:jobId/status` | Статус OCR-задачи | Access JWT |
| GET | `/import/history` | История импортов | Access JWT |

**NATS subjects:**

| Subject | Тип | Описание |
|---|---|---|
| `import.csv.process` | Request/Reply | Запуск обработки CSV-файла |
| `import.ocr.process` | Request/Reply | Запуск OCR-обработки |
| `import.completed` | Publish | Событие об успешном импорте |

**Ключевая бизнес-логика:**

**CSV-импорт (пайплайн):**

1. Принять файл (multipart/form-data), сохранить во временное хранилище.
2. Определить формат: auto-detect разделителя (`,` / `;` / `\t`), кодировки (UTF-8 / Windows-1251), структуры колонок.
3. Поставить задачу в Bull Queue `csv-import`.
4. Воркер: построчно парсить CSV → нормализовать даты и суммы → проверить дедупликацию → батчевая вставка через `bulkCreate` порциями по 500 строк.
5. После завершения: опубликовать `import.completed` → notification-service уведомит пользователя → prediction-service пересчитает прогноз.
6. WebSocket-событие с прогрессом (`import.progress`) транслируется клиенту в реальном времени.

**OCR-импорт (пайплайн):**

1. Принять изображение (JPEG/PNG, макс. 10MB).
2. Предобработка: ресайз до 1920px, нормализация яркости/контраста для улучшения распознавания.
3. Поставить задачу в Bull Queue `ocr-processing`.
4. Воркер: вызов prediction-service (Python) → Google Vision API или локальный Tesseract → парсинг распознанного текста регулярными выражениями (сумма, дата, название магазина).
5. Вернуть распознанные данные клиенту для подтверждения → пользователь корректирует при необходимости → финальное сохранение.

**Поддерживаемые форматы CSV:**

| Банк/Источник | Формат | Кодировка |
|---|---|---|
| Generic (universal) | Дата, Описание, Сумма, Тип | UTF-8 |
| MonoBank (Украина) | Нативный CSV export | UTF-8 |
| PrivatBank (Украина) | XLS/CSV export | Windows-1251 |
| Custom mapping | Настраиваемый маппинг колонок | Авто-определение |

**Зависимости:**

- Bull Queue + Redis (оркестрация фоновых задач)
- `TransactionModule` (сохранение распознанных транзакций)
- `NotificationModule` (уведомление о завершении импорта)
- NATS (события о завершении импорта)
- Временное файловое хранилище (local disk в MVP, S3 — Post-MVP)

### 5.6 PredictionModule

**Ответственность:**

Управление жизненным циклом прогнозов баланса: запрос генерации у Python AI-сервиса, хранение результатов, предоставление данных для дашборда. Является мостом между NestJS-инфраструктурой и Python ML-сервисом.

**Ключевые endpoints:**

| Метод | Путь | Описание | Auth |
|---|---|---|---|
| GET | `/predictions/current` | Последний актуальный прогноз | Access JWT |
| POST | `/predictions/generate` | Запрос пересчёта прогноза вручную | Access JWT |
| GET | `/predictions/history` | История прогнозов | Access JWT |
| GET | `/predictions/:id` | Детали конкретного прогноза | Access JWT |

**NATS subjects:**

| Subject | Тип | Описание |
|---|---|---|
| `prediction.generate` | Request/Reply | Запрос прогноза у Python-сервиса |
| `prediction.completed` | Publish | Прогноз готов — для notification-service |
| `prediction.refresh` | Subscribe | Триггер пересчёта при новых транзакциях |

**Ключевая бизнес-логика:**

**Триггеры пересчёта прогноза:**

Прогноз пересчитывается в следующих случаях:
1. Пользователь явно запросил пересчёт (кнопка в UI).
2. Импорт транзакций завершён (событие `import.completed`).
3. Создана или изменена запись запланированного платежа.
4. Прошёл плановый пересчёт (ежедневно в 03:00 UTC через Bull Delayed Job).

**Кеширование результатов:**

Последний прогноз кешируется в Redis с ключом `prediction:{userId}:{accountId}` и TTL 1 час. При запросе `/predictions/current` данные берутся из Redis, а не из PostgreSQL. Это обеспечивает время ответа < 50ms для самого частого запроса дашборда.

**Структура прогноза (концептуально):**

```
Прогноз для счёта:
- Текущий баланс: $1,240
- Прогнозируемый баланс через 30 дней: $890
- Прогнозируемый баланс через 60 дней: $620
- Прогнозируемый баланс через 90 дней: $310
- Дата потенциального дефицита: 2026-05-12 (риск: высокий)
- Запланированные списания учтённые в прогнозе: [...]
- Точки данных для графика: [...365 значений...]
- Доверительный интервал: ±$85
```

**Зависимости:**

- Python prediction-service (через NATS `prediction.generate`)
- `TransactionModule` (исторические данные для модели)
- `ScheduledPaymentModule` (запланированные платежи для прогноза)
- Redis (кеш прогнозов)
- PostgreSQL (хранение истории прогнозов)
- `NotificationModule` (уведомление о готовности прогноза)

### 5.7 NotificationModule

**Ответственность:**

Единая точка управления всеми уведомлениями системы: Push-уведомления (FCM/APNs), email, и WebSocket real-time события. Модуль потребляет события от других сервисов через NATS и принимает решение о способе и времени доставки.

**Ключевые endpoints:**

| Метод | Путь | Описание | Auth |
|---|---|---|---|
| GET | `/notifications` | Список уведомлений пользователя | Access JWT |
| PATCH | `/notifications/:id/read` | Отметить как прочитанное | Access JWT |
| DELETE | `/notifications/:id` | Удалить уведомление | Access JWT |
| POST | `/notifications/devices` | Регистрация push-токена устройства | Access JWT |
| DELETE | `/notifications/devices/:token` | Отзыв push-токена | Access JWT |
| PATCH | `/notifications/preferences` | Настройки типов уведомлений | Access JWT |

**NATS subjects (потребляемые события):**

| Subject | Событие | Реакция |
|---|---|---|
| `prediction.completed` | Прогноз обновлён | WebSocket push `dashboard:refresh` |
| `prediction.risk.high` | Риск дефицита баланса | Push + Email + WebSocket |
| `import.completed` | Импорт завершён | Push + WebSocket `import:done` |
| `scheduled_payment.due_soon` | Платёж через 3 дня | Push уведомление |
| `user.created` | Новый пользователь | Welcome email |

**Ключевая бизнес-логика:**

- **Приоритизация**: Уведомления о риске дефицита отправляются всеми доступными каналами одновременно (push + email). Информационные уведомления — только push.
- **Do Not Disturb**: Пользователь может настроить временное окно "не беспокоить". Уведомления в этот период откладываются через Bull Delayed Jobs.
- **Rate Limiting**: Не более 3 push-уведомлений в час на пользователя, чтобы избежать notification fatigue.
- **Delivery Tracking**: Статус доставки (sent, delivered, read) хранится в PostgreSQL для аналитики engagement.

**Зависимости:**

- Firebase Admin SDK (FCM для Android push)
- APNs (Apple Push Notification Service для iOS)
- Nodemailer + SMTP (email)
- NATS (подписка на события)
- Bull Queue (delayed notifications, retry)
- PostgreSQL (хранение уведомлений)
- Socket.IO (real-time доставка в WebSocket)

**Ключевые endpoints:**

| Subject | Тип | Описание |
|---|---|---|
| `scheduled_payment.created` | Publish | Триггер пересчёта прогноза |

**Типы периодичности:**

**Авто-обнаружение**: Post-MVP функциональность — автоматическое определение регулярных платежей из истории транзакций через ML-кластеризацию повторяющихся трат.

### 5.9 DashboardModule

**Ответственность:**

Агрегация данных для главного экрана приложения. Не является самостоятельным сервисом — это NestJS-модуль в api-gateway, который оркестрирует запросы к другим модулям и возвращает единый агрегированный ответ.

**Ключевые endpoints:**

| Метод | Путь | Описание | Auth |
|---|---|---|---|
| GET | `/dashboard` | Полные данные дашборда одним запросом | Access JWT |
| GET | `/dashboard/summary` | Краткая сводка (баланс + прогноз) | Access JWT |

**Структура ответа `/dashboard`:**

```json
{
  "account": {
    "id": "uuid",
    "name": "Основной счёт",
    "currency": "USD",
    "currentBalance": 1240.50
  },
  "prediction": {
    "generatedAt": "2026-02-28T10:00:00Z",
    "horizon30": 890.20,
    "horizon60": 620.10,
    "horizon90": 310.80,
    "riskLevel": "medium",
    "riskDate": "2026-05-12",
    "chartData": [{"date": "2026-03-01", "balance": 1200.00}, "..."]
  },
  "recentTransactions": ["...последние 5..."],
  "upcomingPayments": ["...ближайшие 3 платежа..."],
  "unreadNotifications": 2,
  "lastUpdated": "2026-02-28T10:00:00Z"
}
```

**Ключевая бизнес-логика:**

- **Единый запрос**: Дашборд делает параллельные запросы к Account, Prediction и Transaction модулям с использованием `Promise.all` — это минимизирует latency ответа.
- **Кеш дашборда**: Весь агрегированный ответ кешируется в Redis с ключом `dashboard:{userId}` и TTL 2 минуты. Инвалидируется при создании новой транзакции или обновлении прогноза.
- **WebSocket subscription**: После загрузки дашборда клиент подписывается на WebSocket-событие `dashboard:refresh`, которое сигнализирует о необходимости обновить данные без polling.

**Зависимости:**

- `AccountModule`
- `PredictionModule`
- `TransactionModule`
- `ScheduledPaymentModule`
- `NotificationModule`
- Redis (кеш агрегированного ответа)

## 6. Нефункциональные требования — производительность

### 6.1 Целевые показатели

| Метрика | Целевое значение | Измерение | Приоритет |
|---|---|---|---|
| API response time (p50) | < 100ms | Percentile 50 | Обязательно |
| API response time (p95) | < 200ms | Percentile 95 | Обязательно |
| API response time (p99) | < 500ms | Percentile 99 | Желательно |
| Prediction generation | < 2s | End-to-end от запроса до кеша | Обязательно |
| CSV import (1,000 строк) | < 5s | От начала обработки до `import.completed` | Обязательно |
| CSV import (10,000 строк) | < 30s | Фоновая обработка | Желательно |
| OCR processing | < 3s per receipt | От постановки в очередь до результата | Обязательно |
| WebSocket event delivery | < 100ms | От публикации события до получения клиентом | Обязательно |
| Dashboard cold load | < 500ms | Без кеша | Желательно |
| Dashboard warm load | < 50ms | Из Redis-кеша | Обязательно |
| Concurrent users | 10,000+ | Одновременные активные сессии | Обязательно |
| Database transactions table | 100M+ rows | Без деградации query time | Обязательно |

### 6.2 Стратегии достижения показателей

**API Response Time < 200ms (p95):**

- Стратегическое кеширование: баланс счёта, дашборд, прогноз — в Redis.
- Параллельное выполнение независимых запросов через `Promise.all`.
- Database connection pooling через Sequelize (min: 5, max: 20 соединений).
- Индексы на часто используемых полях фильтрации (`account_id`, `date`, `category_id`).
- Cursor-based pagination вместо `OFFSET` для больших таблиц.

**Database 100M+ транзакций:**

- Партиционирование таблицы `transactions` по диапазону дат (по году/кварталу).
- Частичные индексы: индекс только для `deleted_at IS NULL` (активные записи).
- Архивирование транзакций старше 3 лет в отдельную партицию с более медленным storage.
- Read replicas для аналитических запросов (агрегации, статистика).

**Prediction < 2s:**

- Python-сервис предвычисляет прогнозы асинхронно при появлении новых данных, не ожидая клиентского запроса.
- Результат хранится в Redis — клиент получает готовый кеш немедленно.
- Только при явном запросе пересчёта ("обновить прогноз") пользователь ждёт < 2s.

### 6.3 Нагрузочное тестирование

До релиза MVP проводится нагрузочное тестирование с использованием k6:

- **Smoke test**: 5 виртуальных пользователей, 1 минута — базовая работоспособность.
- **Load test**: 1,000 виртуальных пользователей, 10 минут — нормальная нагрузка.
- **Stress test**: 5,000 виртуальных пользователей, пиковая нагрузка — поведение при превышении capacity.

## 7. Нефункциональные требования — безопасность

### 7.1 Аутентификация и авторизация

**Схема токенов:**

```
Access Token:
  - Алгоритм: HS256
  - TTL: 15 минут
  - Payload: { sub: userId, email, tier, iat, exp }
  - Хранение на клиенте: memory (не localStorage, не Cookie без HttpOnly)

Refresh Token:
  - Алгоритм: HS256
  - TTL: 7 дней
  - Хранение: Redis (ключ: refresh:{userId}:{tokenId}, значение: hash)
  - Хранение на клиенте: Secure HttpOnly Cookie или Secure Storage (mobile)
  - Ротация: при каждом использовании старый токен инвалидируется
```

**JwtGuard**: Глобальный guard применяется ко всем маршрутам, кроме явно помеченных `@Public()`. Это fail-secure подход — новый endpoint закрыт по умолчанию.

**Resource Ownership**: Все запросы к ресурсам (счета, транзакции) проверяют, что ресурс принадлежит аутентифицированному пользователю. Проверка происходит на уровне service-слоя через `WHERE account_id = :accountId AND user_id = :userId`.

### 7.2 Защита паролей

- Алгоритм: bcrypt с cost factor 12.
- При cost factor 12 на современном сервере (AWS t3.medium) хеширование занимает ~200-300ms.
- Это делает brute-force атаку экономически нецелесообразной: перебор 1 миллиона паролей займёт ~55 часов на одной машине.
- Пароль никогда не логируется и не сериализуется в ответах API (через `@Exclude()` от `class-transformer`).

### 7.3 Rate Limiting

| Endpoint группа | Лимит | Окно | Реакция при превышении |
|---|---|---|---|
| `POST /auth/login` | 5 requests | 15 минут | 429 Too Many Requests |
| `POST /auth/register` | 3 requests | 1 час | 429 Too Many Requests |
| `POST /auth/refresh` | 10 requests | 1 час | 429 Too Many Requests |
| Все `/api/*` | 100 requests | 1 минута | 429 Too Many Requests |
| `POST /import/ocr` | 10 requests | 1 час | 429 Too Many Requests |

Rate limiting реализуется через `@nestjs/throttler` с Redis store — счётчики хранятся в Redis, что обеспечивает консистентность при нескольких инстансах api-gateway.

### 7.4 Валидация входных данных

- `class-validator` и `class-transformer` применяются ко всем DTO через глобальный `ValidationPipe`.
- `whitelist: true` — все свойства, не описанные в DTO, автоматически удаляются из запроса.
- `forbidNonWhitelisted: true` — запрос с неизвестными полями возвращает 400.
- Максимальный размер тела запроса: 1MB (кроме `/import/*` endpoints: 10MB).
- Все string-поля проходят `@MaxLength()` валидацию для предотвращения атак через длинные строки.

### 7.5 Защита от SQL-инъекций

Sequelize использует parameterized queries для всех операций с базой данных. Прямое конкатенирование пользовательского ввода в SQL-запросы — строго запрещено в code review checklist.

### 7.6 CORS

```typescript
// Допустимые origins в production
app.enableCors({
  origin: ['https://app.anticifi.com', 'https://anticifi.com'],
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
  credentials: true,
  maxAge: 86400,
});
```

В development разрешён `*` для удобства разработки.

### 7.7 HTTPS

- Все production endpoints доступны только через HTTPS (TLS 1.2+).
- HTTP → HTTPS redirect через Nginx.
- SSL-сертификаты: Let's Encrypt (auto-renewal через Certbot).
- HSTS header: `max-age=31536000; includeSubDomains`.

### 7.8 Шифрование данных

- **At Rest**: PostgreSQL TDE (Transparent Data Encryption) через pg_crypto или шифрование на уровне файловой системы (LUKS на Linux). Особенно критично для столбцов с суммами транзакций.
- **In Transit**: TLS 1.2+ для всех внешних соединений. Внутренние NATS/Redis-соединения шифруются TLS в production (в MVP — в рамках private network).
- **Sensitive Fields**: Токены устройств (FCM/APNs) шифруются перед хранением в PostgreSQL через application-level encryption (AES-256).

### 7.9 GDPR Compliance

| Требование GDPR | Реализация |
|---|---|
| Право на доступ (Art. 15) | `GET /users/export` — полный JSON-экспорт данных |
| Право на удаление (Art. 17) | `DELETE /users/account` — soft delete + hard delete через 30 дней |
| Право на портативность (Art. 20) | Экспорт в machine-readable JSON |
| Минимизация данных | Сбор только необходимых данных, нет рекламного трекинга |
| Явное согласие | Checkbox при регистрации, хранение timestamp согласия |
| Data Breach Notification | Процедура уведомления в течение 72 часов (операционная) |

## 8. Нефункциональные требования — масштабируемость

### 8.1 Горизонтальное масштабирование

Все backend-сервисы спроектированы как stateless — они не хранят состояние в памяти процесса. Всё состояние вынесено в Redis (сессии, кеш) и PostgreSQL (персистентные данные).

Это позволяет запустить N инстансов любого сервиса за Nginx load balancer без изменения кода:

```
Nginx (upstream round-robin)
  ├── api-gateway instance 1
  ├── api-gateway instance 2
  └── api-gateway instance N
```

**WebSocket и горизонтальное масштабирование**: Socket.IO использует Redis Adapter для синхронизации комнат между инстансами. Событие, опубликованное в инстансе 1, будет доставлено клиентам, подключённым к инстансу 2.

### 8.2 Database Scaling

| Стратегия | Описание | Когда применяется |
|---|---|---|
| Connection Pooling | Sequelize pool (max 20) | MVP |
| Read Replicas | Streaming replication для чтения | При росте read-нагрузки |
| Table Partitioning | Партиционирование `transactions` по дате | При > 10M записей |
| Archiving | Перенос старых данных в cold storage | При > 100M записей |
| PgBouncer | Connection pooler перед PostgreSQL | При > 100 инстансов app |

### 8.3 Redis Cluster Mode

В production Redis переводится в Cluster Mode с минимум 3 master-нодами и 3 replica-нодами. Это обеспечивает:
- Горизонтальное шардирование данных по нодам.
- Автоматический failover при падении master-ноды.
- Продолжение работы при потере до 50% нод.

### 8.4 NATS Cluster

NATS развёртывается как кластер из 3 нод (нечётное количество для quorum). Это обеспечивает:
- Автоматическое переподключение клиентов при падении ноды.
- Репликацию JetStream-сообщений для at-least-once delivery.
- Продолжение работы при потере 1 ноды из 3.

### 8.5 CDN для статических ресурсов

Статические ресурсы (иконки, шрифты, изображения категорий) размещаются на CDN (Cloudflare в MVP). Это снижает нагрузку на Nginx и уменьшает latency для географически удалённых пользователей.

### 8.6 Capacity Planning (MVP → Scale)

| Фаза | Пользователи | Транзакций/день | Инфраструктура |
|---|---|---|---|
| MVP | 0 — 1,000 | ~10,000 | 1 VPS (4 CPU, 8GB RAM), Docker Compose |
| Early Growth | 1,000 — 10,000 | ~100,000 | 2-3 VPS, Docker Swarm или K8s |
| Scale | 10,000+ | ~1,000,000 | Kubernetes, managed PostgreSQL (RDS), ElastiCache |

## 9. Ограничения и допущения

### 9.1 Технические ограничения

| Ограничение | Описание |
|---|---|
| OCR точность | Google ML Kit Text Recognition обеспечивает ~85-90% точность на стандартных кассовых чеках. Нестандартные форматы требуют ручной корректировки. |
| ML точность | Для новых пользователей (< 1 месяц истории) прогноз будет основан только на указанных запланированных платежах, без ML-компоненты. |
| Одновременный импорт | Пользователь может иметь не более 1 активной задачи CSV-импорта одновременно. |
| Размер CSV | Максимальный размер загружаемого CSV-файла — 10MB (~100,000 строк). Большие файлы должны быть разбиты. |
| Размер изображения для OCR | Максимальный размер — 10MB. Рекомендуемый формат: JPEG. |
| Free Tier | 1 финансовый счёт. Хранение истории — неограничено. |

### 9.2 Допущения

- Пользователи имеют стабильное интернет-соединение для синхронизации данных. Офлайн-режим поддерживает только просмотр кешированных данных.
- CSV-файлы банковских выписок имеют стандартизированный формат. Нестандартные форматы требуют ручного маппинга колонок.
- Курсы валют не конвертируются автоматически — все транзакции счёта ведутся в одной валюте.
- Пользователь самостоятельно категоризирует транзакции в MVP. Авто-категоризация — Post-MVP.
- Система рассчитана на B2C использование (физические лица), не на корпоративное.

### 9.3 Внешние зависимости

| Сервис | Назначение | Fallback при недоступности |
|---|---|---|
| Firebase FCM | Android push-уведомления | Graceful skip, повтор при восстановлении |
| Apple APNs | iOS push-уведомления | Graceful skip, повтор при восстановлении |
| SMTP Server | Email-уведомления | Queue с retry, email потенциально задержан |
| Google ML Kit | OCR (on-device) | Ошибка задачи, пользователь вводит данные вручную |

## Epic 1: Auth — Аутентификация и управление сессиями

Эпик охватывает весь жизненный цикл идентификации пользователя: создание аккаунта, вход, восстановление доступа, биометрику и управление активными сессиями. Безопасность аутентификации является основой для всех остальных эпиков.

---

### US-001: Регистрация через email и пароль

**Как** гость, **я хочу** зарегистрироваться в приложении через email и пароль, **чтобы** создать личный аккаунт и начать пользоваться AnticiFi.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Форма регистрации содержит поля: email, пароль, подтверждение пароля; все поля обязательны
- Пароль должен содержать минимум 8 символов, включая хотя бы одну цифру и одну заглавную букву; при несоответствии отображается понятное сообщение об ошибке
- После успешной регистрации на указанный email отправляется письмо с ссылкой для подтверждения; аккаунт без подтверждения ограничен в функциях
- Email проверяется на уникальность: если адрес уже зарегистрирован, система возвращает ошибку без раскрытия факта существования аккаунта (сообщение: "Если этот email свободен, вы получите письмо")
- После подтверждения email пользователь автоматически авторизуется и перенаправляется на Dashboard

**API:** `POST /api/v1/auth/register`, `POST /api/v1/auth/verify-email`

---

### US-002: Вход в систему с JWT-токенами

**Как** пользователь, **я хочу** войти в приложение через email и пароль, **чтобы** получить доступ к своим финансовым данным.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Форма входа содержит поля email и пароль; поддерживается автозаполнение браузером/менеджером паролей
- При успешной аутентификации система выдаёт пару токенов: access token (TTL 15 минут) и refresh token (TTL 30 дней), refresh token сохраняется в httpOnly cookie
- После 5 неудачных попыток входа подряд аккаунт временно блокируется на 15 минут; пользователь получает уведомление на email
- Access token автоматически обновляется через refresh token без повторного ввода пароля; при истечении refresh token пользователь перенаправляется на экран входа
- При входе с нового устройства или IP-адреса на email отправляется уведомление безопасности

**API:** `POST /api/v1/auth/login`, `POST /api/v1/auth/refresh`

---

### US-003: Сброс пароля через email

**Как** пользователь, **я хочу** восстановить доступ к аккаунту через email, **чтобы** не потерять свои данные при забытом пароле.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Форма запроса сброса принимает только email; ответ системы одинаков при существующем и несуществующем email (защита от перебора)
- Письмо со ссылкой для сброса отправляется в течение 60 секунд; ссылка содержит одноразовый токен и действует 1 час
- Токен сброса инвалидируется немедленно после использования; повторное использование той же ссылки возвращает ошибку
- Новый пароль проходит те же валидации, что и при регистрации; после успешного сброса все существующие сессии пользователя отзываются
- Пользователь получает подтверждающее email-уведомление об успешной смене пароля

**API:** `POST /api/v1/auth/forgot-password`, `POST /api/v1/auth/reset-password`

---

### US-004: Биометрический вход на мобильном устройстве

**Как** пользователь мобильного приложения, **я хочу** входить в приложение через отпечаток пальца или Face ID, **чтобы** не вводить пароль каждый раз и обеспечить быстрый доступ к данным.

**Приоритет:** Should

**Критерии приёмки:**
- Биометрический вход доступен только на устройствах, поддерживающих Touch ID или Face ID; на несовместимых устройствах опция скрыта
- Биометрия активируется пользователем вручную в настройках после успешного входа через пароль; приватный ключ хранится в Secure Enclave / Keystore устройства, на серверы не передаётся
- При успешной биометрической аутентификации приложение получает новую пару токенов без ввода пароля
- При 3 неудачных биометрических попытках подряд приложение откатывается к вводу пароля
- При смене биометрических данных на устройстве (добавлен новый отпечаток) биометрический вход автоматически деактивируется и требует повторной настройки

**API:** `POST /api/v1/auth/biometric/challenge`, `POST /api/v1/auth/biometric/verify`

---

### US-005: Управление сессиями

**Как** пользователь, **я хочу** видеть все активные сессии и завершать любую из них, **чтобы** контролировать безопасность своего аккаунта.

**Приоритет:** Should

**Критерии приёмки:**
- Раздел "Активные сессии" отображает список: устройство (название, тип), последнее местоположение (город, страна по IP), дата и время последней активности
- Текущая сессия выделена в списке и не может быть удалена через этот интерфейс (только через кнопку "Выйти")
- Пользователь может завершить любую отдельную стороннюю сессию; refresh token этой сессии немедленно отзывается
- Кнопка "Выйти со всех устройств" отзывает все refresh токены кроме текущего; пользователь остаётся залогиненным на текущем устройстве
- Выход из текущей сессии очищает локальное хранилище токенов и перенаправляет на экран входа

**API:** `GET /api/v1/auth/sessions`, `DELETE /api/v1/auth/sessions/:id`, `DELETE /api/v1/auth/sessions`

## Epic 2: Import — Импорт и категоризация транзакций

Эпик охватывает все способы добавления финансовых данных в систему: автоматический импорт из банковских выписок, OCR-сканирование чеков, ручной ввод и интеллектуальную обработку транзакций. Качество импорта напрямую влияет на точность предсказаний Oracle.

---

### US-010: Импорт CSV из банковской выписки

**Как** пользователь, **я хочу** загрузить CSV-файл банковской выписки, **чтобы** быстро добавить историю транзакций без ручного ввода.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Интерфейс поддерживает drag-and-drop и выбор файла; принимаются файлы .csv и .xlsx до 10 МБ
- После загрузки система отображает предпросмотр первых 10 строк с возможностью настройки маппинга колонок (дата, описание, сумма, валюта) перед финальным импортом
- Система автоматически определяет формат даты (DD.MM.YYYY, YYYY-MM-DD, MM/DD/YYYY) и символ разделителя (запятая, точка с запятой)
- Прогресс импорта отображается в реальном времени (обработано X из N строк); при ошибке в конкретной строке она пропускается и добавляется в отчёт об ошибках
- После завершения импорта отображается итоговая сводка: добавлено, пропущено (дубликаты), ошибки; пользователь может скачать отчёт об ошибках

**API:** `POST /api/v1/imports/csv`, `GET /api/v1/imports/:id/status`

---

### US-011: OCR-сканирование бумажных чеков

**Как** пользователь, **я хочу** сфотографировать бумажный чек, **чтобы** добавить транзакцию без ручного ввода данных.

**Приоритет:** Should

**Критерии приёмки:**
- Мобильное приложение предоставляет камеру с рамкой-подсказкой для правильного кадрирования чека; поддерживается также загрузка из галереи
- OCR-движок извлекает: дату, итоговую сумму, наименование продавца; успешность распознавания не менее 85% для стандартных чеков
- Перед сохранением система показывает форму с распознанными данными для проверки и редактирования пользователем
- Изображение чека сохраняется как вложение к транзакции и доступно для просмотра из истории
- При низком качестве изображения (размытость, недостаточное освещение) система запрашивает повторную съёмку с подсказкой по улучшению

**API:** `POST /api/v1/imports/receipt`

---

### US-012: Ручной ввод транзакции

**Как** пользователь, **я хочу** вручную добавить транзакцию, **чтобы** учесть операции, которые невозможно импортировать автоматически (наличные, переводы).

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Форма содержит обязательные поля: сумма, дата, тип (доход/расход); необязательные: описание, категория, счёт, метки
- Сумма поддерживает ввод в любой из подключённых валют с автоматической конвертацией по курсу на дату транзакции
- Дата по умолчанию — текущий день; выбор через нативный date picker; разрешён ввод дат в прошлом и будущем (для запланированных платежей)
- После сохранения транзакция немедленно отображается в истории и учитывается в расчётах баланса и предсказаниях
- Пользователь может дублировать существующую транзакцию как шаблон для нового ввода

**API:** `POST /api/v1/transactions`

---

### US-013: Дедупликация транзакций

**Как** пользователь, **я хочу** чтобы система автоматически обнаруживала дублирующиеся транзакции при импорте, **чтобы** избежать двойного учёта одних и тех же операций.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Система вычисляет `transactionHash` на основе комбинации: дата, сумма, описание (нормализованное); транзакция с существующим хешем помечается как дубликат
- При обнаружении потенциального дубликата пользователю показывается сравнение двух транзакций (существующая vs новая) с вариантами: "Пропустить", "Сохранить как новую", "Заменить существующую"
- Порог схожести для нечёткого сопоставления (fuzzy match) настраиваем: 90% по умолчанию (одинаковая сумма + дата ±1 день + похожее описание)
- В отчёте об импорте явно указывается количество пропущенных дубликатов
- Пользователь может просмотреть все автоматически пропущенные дубликаты и вручную решить для каждого

**API:** `POST /api/v1/transactions/check-duplicate`

---

### US-014: История импортов со статусами

**Как** пользователь, **я хочу** видеть историю всех импортов и их текущий статус, **чтобы** отслеживать прогресс и находить ошибки загрузки данных.

**Приоритет:** Should

**Критерии приёмки:**
- Список импортов содержит: дату и время, источник (CSV/OCR/ручной), статус (pending / processing / completed / failed), количество транзакций (добавлено / пропущено / ошибок)
- Для завершённых импортов доступна кнопка "Посмотреть детали" с разбивкой по строкам и описанием ошибок
- Для упавших импортов отображается причина ошибки и кнопка "Повторить попытку"
- Статус обновляется в реальном времени через WebSocket или polling каждые 3 секунды для активных импортов
- Пользователь может удалить запись об импорте; при этом транзакции, добавленные этим импортом, остаются в системе (не удаляются каскадно)

**API:** `GET /api/v1/imports`, `GET /api/v1/imports/:id`, `DELETE /api/v1/imports/:id`

---

### US-015: Автоопределение категории транзакции

**Как** пользователь, **я хочу** чтобы транзакции автоматически получали категорию на основе описания, **чтобы** не тратить время на ручную категоризацию каждой операции.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- ML-модель определяет категорию по описанию транзакции; точность автоматической категоризации не менее 75% для стандартных транзакций (супермаркеты, АЗС, рестораны, подписки)
- Предсказанная категория отображается с иконкой и уровнем уверенности; если уверенность ниже 60%, категория отмечается как "требует проверки"
- Пользователь может изменить категорию одним тапом; исправление обучает персональную модель (feedback loop)
- Система запоминает пользовательские правила: "Всё от NETFLIX → Подписки"; правила имеют приоритет над ML-предсказанием
- Категории организованы в двухуровневую иерархию: "Еда" → "Рестораны", "Продукты", "Доставка"

**API:** `POST /api/v1/ml/categorize`, `PUT /api/v1/transactions/:id/category`

## Epic 3: Oracle — Предсказания и аналитика

Oracle — ключевая ценность AnticiFi. Эпик охватывает все сценарии взаимодействия пользователя с предиктивным AI-движком: от прямых запросов баланса до алертов и ретроспективного анализа точности.

---

### US-020: Предсказание баланса на конкретную дату

**Как** пользователь, **я хочу** узнать ожидаемый баланс на любую будущую дату, **чтобы** планировать крупные расходы и принимать финансовые решения заранее.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Интерфейс предоставляет date picker для выбора будущей даты; поддерживается ввод дат до 12 месяцев вперёд
- Предсказание учитывает: историю расходов за последние 90 дней, запланированные регулярные платежи, сезонные паттерны (повышение расходов в декабре, отпуска)
- Результат отображается в виде диапазона: "Ожидаемый баланс: от X до Y"; точечное предсказание выделено, диапазон отображает 80% доверительный интервал
- Рядом с предсказанием показывается топ-3 фактора, влияющих на результат (например: "Аренда −50 000 ₽", "Подписки −5 200 ₽", "Обычные расходы −32 000 ₽")
- Предсказание пересчитывается при изменении исходных данных (новые транзакции, изменение запланированных платежей) без необходимости повторного запроса

**API:** `GET /api/v1/oracle/predict?date=YYYY-MM-DD`

---

### US-021: Запрос на естественном языке о финансовом будущем

**Как** пользователь, **я хочу** задавать вопросы о своих финансах на обычном русском языке, **чтобы** получать ответы без необходимости разбираться в интерфейсе.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Текстовое поле принимает вопросы на русском языке; примеры поддерживаемых вопросов: "Хватит ли мне денег до зарплаты?", "Когда закончатся деньги если я потрачу 30 000 на отпуск?", "Сколько я трачу на кафе каждый месяц?"
- NLP-движок интерпретирует вопрос и формирует структурированный запрос к предиктивной модели; ответ содержит прямой ответ + краткое объяснение логики
- Время ответа не превышает 3 секунд для 95% запросов; пользователь видит индикатор загрузки
- Если вопрос не может быть понят или выходит за рамки финансовой тематики, система вежливо объясняет ограничения и предлагает переформулировку
- История последних 20 вопросов сохраняется и доступна для повторного запроса одним нажатием

**API:** `POST /api/v1/oracle/ask`

---

### US-022: Уровень уверенности предсказания

**Как** пользователь, **я хочу** видеть насколько система уверена в своём предсказании, **чтобы** понимать, насколько можно доверять прогнозу при принятии решений.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Уровень уверенности отображается в процентах (0–100%) и визуально (цветовая шкала: зелёный > 75%, жёлтый 50–75%, красный < 50%) рядом с каждым предсказанием
- Тултип при наведении объясняет факторы, снижающие уверенность: "Мало данных за последние 30 дней", "Высокая вариабельность расходов", "Запланированный платёж без истории"
- Уверенность автоматически снижается с увеличением горизонта предсказания: предсказания на 1 неделю имеют более высокую уверенность, чем на 6 месяцев; это явно показывается пользователю
- Система не показывает предсказания с уверенностью ниже 20%; вместо этого отображается сообщение "Недостаточно данных для надёжного прогноза" с рекомендацией добавить больше транзакций
- Уровень уверенности возвращается в ответе API как числовое поле `confidence` (0.0–1.0) и словесное описание `confidence_label`

**API:** поле `confidence` в ответах `GET /api/v1/oracle/predict`, `POST /api/v1/oracle/ask`

### US-023: Анализ трендов расходов

**Как** пользователь, **я хочу** видеть тренды своих расходов по категориям, **чтобы** понять, где я трачу больше чем планировал и в каком направлении движутся мои финансы.

**Приоритет:** Should

**Критерии приёмки:**
- Раздел "Тренды" отображает графики изменения расходов по выбранным категориям за последние 3, 6 или 12 месяцев
- Для каждой категории система рассчитывает: среднемесячный расход, изменение по сравнению с предыдущим периодом (+/−%), линейный тренд (растёт/падает/стабильно)
- Аномальные месяцы выделяются на графике с пояснением ("Декабрь: +85% — праздничные расходы")
- Система генерирует текстовый инсайт для топ-3 трендов: "Ваши расходы на рестораны выросли на 23% за 3 месяца. При текущем темпе это добавит 4 200 ₽/мес к вашему бюджету"
- Пользователь может сравнить два периода (например, Q3 2025 vs Q4 2025) на одном графике

**API:** `GET /api/v1/oracle/trends`

### US-024: Алерт при падении предсказанного баланса ниже порога

**Как** пользователь, **я хочу** получить предупреждение заранее, когда прогнозируемый баланс опустится ниже установленного мной порога, **чтобы** успеть принять меры до возникновения финансовых проблем.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Пользователь задаёт минимальный порог баланса (например, 10 000 ₽) в настройках; порог задаётся отдельно для каждого счёта или суммарно по всем счетам
- Система ежедневно пересчитывает предсказания и отправляет уведомление при обнаружении, что в течение следующих 30 дней ожидается падение ниже порога
- Уведомление содержит: ожидаемую дату падения, ожидаемую сумму, и топ-3 причины (крупные запланированные платежи или тренды)
- Повторное уведомление по той же ситуации отправляется не чаще 1 раза в 7 дней, чтобы не раздражать пользователя
- Уведомление снимается автоматически, если ситуация улучшилась (пользователь пополнил счёт или ситуация изменилась)

**API:** `POST /api/v1/alerts`, `GET /api/v1/alerts`, связано с US-040

### US-025: Сравнение предсказаний с реальными данными

**Как** пользователь, **я хочу** видеть насколько точными были прошлые предсказания системы, **чтобы** оценивать достоверность текущих прогнозов.

**Приоритет:** Could

**Критерии приёмки:**
- Раздел "Точность прогнозов" отображает историю предсказаний: дата предсказания, горизонт (на сколько дней вперёд), предсказанное значение, фактическое значение, отклонение в рублях и процентах
- Сводная метрика точности отображается как MAPE (Mean Absolute Percentage Error) за последние 30/90 дней
- График сопоставляет предсказанный и реальный баланс на одной шкале; зоны расхождения закрашены
- Система автоматически анализирует причины крупных расхождений и отображает пояснение: "Предсказание от 15 января расходилось на 18% из-за внеплановой покупки 22 января"
- Данные для сравнения доступны только за периоды, когда пользователь уже использовал AnticiFi (нет ретроспективных предсказаний)

**API:** `GET /api/v1/oracle/accuracy`

- Полнотекстовый поиск по описанию транзакции работает в реальном времени при вводе (debounce 300ms)
- Список поддерживает бесконечную прокрутку; первая загрузка — 20 элементов, подгрузка по 20 при скролле

### US-034: Сводные карточки месяца

**Как** пользователь, **я хочу** видеть ключевые показатели текущего месяца в виде компактных карточек, **чтобы** мгновенно оценивать финансовую картину без детального анализа.

- Карточки интерактивны: нажатие открывает детальный вид соответствующего раздела

### US-040: Push-уведомление о низком предсказанном балансе

- Уведомление не отправляется если пользователь отключил этот тип в настройках (US-044)
- Повторное уведомление по той же ситуации отправляется не чаще 1 раза в 3 дня
- При получении уведомления в системе создаётся запись в журнале уведомлений (в приложении), видимая даже без push-разрешений

### US-041: Напоминание о предстоящем запланированном платеже

**Критерии приёмки:**

- Уведомление отправляется в предпочтительное время пользователя (настраивается в US-044); по умолчанию — 9:00 по часовому поясу устройства
- После фиксации транзакции (платёж прошёл) уведомление автоматически помечается как выполненное в журнале
- Если платёж пропущен (дата прошла, транзакция не зафиксирована), отправляется повторное уведомление с пометкой "Просроченный платёж"

- Нажатие на любой элемент сводки открывает соответствующий раздел приложения

### US-043: Алерт о нетипичных расходах

**Как** пользователь, **я хочу** получать уведомление при обнаружении необычно высоких трат, **чтобы** вовремя замечать потенциально мошеннические операции или выход за рамки привычного бюджета.

**Приоритет:** Should

**Критерии приёмки:**
- Транзакция считается аномальной если: сумма превышает 3σ от среднего по данной категории за последние 90 дней, или транзакция в новой для пользователя категории на сумму >5 000 ₽

- Пользователь может отметить транзакцию как "Всё в порядке" (обучает модель) или "Сообщить о проблеме" (заглушка для будущей интеграции с банком)
- Порог аномальности настраивается: выключено / только крупные (>10 000 ₽) / все аномалии
- Алерты об аномалиях не отправляются для транзакций из импорта CSV (только для новых данных в реальном времени)

**API:** фоновый процесс анализа, `POST /api/v1/notifications/anomaly`

### US-044: Управление предпочтениями уведомлений

- Для каждого типа настраивается канал доставки: push, email, оба, ни один
- Настраиваются "тихие часы" (например, с 23:00 до 8:00): в этот период уведомления накапливаются и отправляются пакетом в конце тихих часов
- Предпочтения синхронизируются между устройствами; при удалении приложения и повторной установке настройки восстанавливаются
- Для каждого уведомления в журнале приложения доступна кнопка "Больше не показывать такие" для быстрого отключения

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Форма добавления счёта содержит поля: название (обязательно), тип (дебетовый, кредитный, накопительный, наличные), банк (выбор из справочника или ручной ввод), валюта, начальный баланс
- Для кредитных счетов дополнительные поля: кредитный лимит, дата ежемесячного платежа
- Редактирование счёта не изменяет историю привязанных транзакций; изменение начального баланса пересчитывает текущий баланс
- При удалении счёта пользователю предлагается выбор: удалить только счёт (транзакции перемещаются в "Без счёта") или удалить счёт вместе с транзакциями (требует подтверждения с явным предупреждением)
- Счёт можно архивировать (скрыть из активных, но сохранить историю) без полного удаления

**API:** `GET /api/v1/accounts`, `POST /api/v1/accounts`, `PUT /api/v1/accounts/:id`, `DELETE /api/v1/accounts/:id`

### US-053: Управление регулярными платежами

**Как** пользователь, **я хочу** управлять списком регулярных платежей (аренда, подписки, кредиты), **чтобы** система учитывала их в предсказаниях баланса.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Форма добавления регулярного платежа: название, сумма, валюта, периодичность (ежедневно, еженедельно, ежемесячно, ежегодно), дата следующего платежа, счёт списания, категория
- Периодичность поддерживает кастомные интервалы: "каждые N дней", "каждый N-й день месяца", "каждый последний рабочий день месяца"
- Платёж с прошедшей датой автоматически вычисляет следующую дату на основе периодичности; исторические вхождения не создаются автоматически
- Изменение суммы или периодичности применяется только к будущим вхождениям; прошлые транзакции не изменяются
- Список регулярных платежей отображается с суммарной нагрузкой на месяц (сумма всех ежемесячных эквивалентов)

**API:** `GET /api/v1/scheduled-payments`, `POST /api/v1/scheduled-payments`, `PUT /api/v1/scheduled-payments/:id`, `DELETE /api/v1/scheduled-payments/:id`

### US-054: Экспорт данных (CSV / PDF)

**Как** пользователь, **я хочу** экспортировать историю транзакций в CSV или PDF, **чтобы** использовать данные в сторонних программах или предоставить в налоговую.

**Приоритет:** Could

**Критерии приёмки:**
- Экспорт настраивается по: диапазону дат, счётам (один или все), форматам (CSV, PDF); фильтры аналогичны US-031
- CSV-файл содержит колонки: ID, дата, описание, категория, сумма, валюта, счёт, тип; первая строка — заголовки
- PDF-отчёт содержит: логотип AnticiFi, имя пользователя, период отчёта, сводную таблицу по категориям, детальную таблицу транзакций; оформлен в читаемом виде для печати
- Генерация PDF для больших периодов (>1000 транзакций) выполняется асинхронно; пользователь получает ссылку для скачивания по email или push-уведомлением
- Ссылка для скачивания действительна 24 часа; файл удаляется с серверов после скачивания или по истечении срока

**API:** `POST /api/v1/export`, `GET /api/v1/export/:id/download`

### US-055: Удаление аккаунта с очисткой данных

**Как** пользователь, **я хочу** полностью удалить свой аккаунт и все связанные данные, **чтобы** реализовать своё право на забвение согласно требованиям GDPR/152-ФЗ.

**Приоритет:** Must (MVP)

**Критерии приёмки:**
- Процесс удаления состоит из 3 шагов: причина удаления (опционально), предложение скачать данные перед удалением (PDF/CSV экспорт), финальное подтверждение вводом пароля
- После подтверждения система немедленно деактивирует аккаунт (вход невозможен) и ставит задачу на полную очистку данных в течение 30 дней (стандарт GDPR)
- Удаляются все персональные данные: профиль, транзакции, счета, импорты, токены, уведомления; анонимизированные агрегированные данные могут быть сохранены для аналитики (без связи с личностью)
- Пользователь получает email-подтверждение о начале процедуры удаления и финальное подтверждение о завершении через 30 дней
- В течение 7 дней после запроса пользователь может отменить удаление, обратившись в поддержку; после 7 дней отмена невозможна

**API:** `POST /api/v1/account/delete-request`, `DELETE /api/v1/account`
